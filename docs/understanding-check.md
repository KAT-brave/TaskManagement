## Q1. インターネットとは

「インターネット」と「Webブラウザで見るホームページ」は同じものですか？違うとすれば、どう違いますか？


## A1.
「インターネット」と「Webブラウザで見るホームページ」は同じものではありません。
インターネットは、世界中のコンピューターをつなぐ大きなネットワークです。 ホームページは、そのインターネットを使って見られるサービスの一つです。
例えば、インターネットではホームページを見る以外にも、メール、チャット、動画視聴、オンラインゲームなどができます。
つまり、

インターネット = 通信の仕組み
ホームページ = インターネット上で利用できるサービスの一つ

という違いがあります。

## Q2. IPアドレスとポート番号

このプロジェクトではフロントエンドを `localhost:5173`、バックエンドを `localhost:8080` で起動しています。

- `localhost`とは何を指しますか？
- `:5173` や `:8080` のようなポート番号はなぜ必要ですか？

## A2.
localhost は、自分のパソコン自身を指します。 今回であれば、インターネット上の外部サーバーではなく、自分のMac上でフロントエンドとバックエンドを起動して確認しているという意味です。
:5173 や :8080 はポート番号で、同じ localhost の中で、どのアプリにアクセスするかを区別するために必要です。
今回のプロジェクトでは、

localhost:5173 → Reactのフロントエンド
localhost:8080 → Spring Bootのバックエンド

という役割になっています。
同じ自分のパソコンの中で複数のアプリが動いているため、ポート番号を使ってアクセス先を分けています。 たとえるなら、localhost が建物で、:5173 や :8080 は部屋番号のようなものだと理解しています。

## Q3. DNS

ブラウザのアドレスバーに

`https://www.example.com`

と入力してからページが表示されるまでに、DNSはどのような役割を果たしますか？


## A3.
DNSは、入力したサイト名を、実際の接続先であるIPアドレスに変換する役割を持っています。
ブラウザに

https://www.example.com

と入力しても、コンピューターはその文字だけでは接続先が分かりません。
そこでDNSが、

www.example.com はどのIPアドレスか

を調べます。
その結果をもとに、ブラウザが正しいサーバーへアクセスし、ページが表示されます。
つまりDNSは、人間が分かりやすいサイト名を、コンピューターが接続できる住所に変換する仕組みだと理解しています。

## Q4. HTTPとは

HTTPとは何の略で、どのような役割を持つものですか？

「プロトコル」という言葉の意味も含めて説明してください。


## A4.
HTTPは、HyperText Transfer Protocol の略です。 Webページなどのデータを、ブラウザとサーバーの間でやり取りするためのルールです。
「プロトコル」とは、通信するときの決まりごとのことです。 人同士の会話でも共通の言葉が必要なように、コンピューター同士も同じルールで通信する必要があります。
HTTPでは、ブラウザがサーバーに

このページを見せてください

とリクエストを送り、サーバーがHTMLや画像などを返します。
HTTPはWebページを表示するための通信ルールだと理解しています。

## Q5. HTTPとHTTPSの違い

HTTPとHTTPSの違いは何ですか？
なぜ現在のWebサイトはほぼすべてHTTPSを使用していますか？

## A5.
HTTPとHTTPSの違いは、通信が暗号化されているかどうかです。
HTTPは通信内容が暗号化されていないため、第三者に見られたり、改ざんされたりするリスクがあります。 一方、HTTPSは通信内容が暗号化されるため、ログイン情報や個人情報をより安全にやり取りできます。
現在のWebサイトでHTTPSが多く使われているのは、利用者の情報を守り、安全にサイトを使えるようにするためです。
つまり、

HTTP  = 暗号化なし
HTTPS = 暗号化あり

という違いだと理解しています。

## Q6. リクエストとレスポンス

Webにおける「リクエスト」と「レスポンス」の関係を、日常生活の例えを使って説明してください。


## A6.
リクエストは、ブラウザからサーバーに対して「この情報をください」とお願いすることです。 レスポンスは、そのお願いに対してサーバーが「はい、これです」と返すことです。
日常生活で例えると、レストランで店員さんに注文するのがリクエストで、注文した料理が運ばれてくるのがレスポンスです。
Webでも同じように、ブラウザが
「このページを表示してください」
とサーバーにリクエストを送り、サーバーがHTMLや画像などのデータをレスポンスとして返すことで、ページが表示されます。

## Q7. HTTPメソッド

このアプリのAPIでは以下を使っています。

- GET
- POST
- PUT
- DELETE
- PATCH

これらの「HTTPメソッド」とは何ですか？

日常生活の例えで、それぞれの違いを説明してください。

## A7.

HTTPメソッドとは、ブラウザやフロントエンドがサーバーに対して「何をしたいか」を伝えるための命令の種類です。
日常生活でタスク管理アプリを「ノート」に例えると以下になります。

GET    = ノートを見る
POST   = ノートに新しいタスクを書き足す
PUT    = 既存のタスクを全体的に書き直す
PATCH  = 既存のタスクの一部だけ直す
DELETE = タスクを消す

今回のアプリでいうと、GET はカード一覧を見る処理、POST は新しいカードを作る処理、PUT や PATCH はカードの内容を変更する処理、DELETE はカードを削除する処理です。
つまり、HTTPメソッドは「データに対して、見る・作る・直す・消すのどれをしたいか」をサーバーに伝えるためのものだと理解しています。

## Q8. HTTPステータスコード

サーバーからの返事には「ステータスコード」という3桁の数字が含まれます。

以下のコードはそれぞれ何を意味しますか？

- 200
- 201
- 400
- 404
- 500

## A8.
ステータスコードは、サーバーからブラウザやフロントエンドに返される処理結果を表す3桁の数字です。 簡単に言うと、「リクエストが成功したのか、失敗したのか」を知らせるものです。
日常生活で例えると、お店で注文したときの返事のようなものです。

200 = 正常に処理できました

注文した商品を問題なく受け取れた状態です。 例：カード一覧を取得できた。

201 = 新しく作成できました

注文したものが新しく作られた状態です。 例：新しいカードを登録できた。

400 = リクエスト内容に問題があります

注文内容に不備があって、店員さんが対応できない状態です。 例：タイトル未入力のままカード作成しようとした。

404 = 指定されたものが見つかりません

注文した商品やページが存在しない状態です。 例：存在しないカードIDにアクセスした。

500 = サーバー側でエラーが起きました

注文内容は問題ないが、お店側の機械トラブルなどで対応できない状態です。 例：プログラムやサーバー内部で予期しないエラーが起きた。

つまり、ステータスコードはサーバーが処理結果を数字で伝えるためのものだと理解しています。

## Q9. JSONとは

フロントエンドとバックエンド間でデータをやり取りする際に「JSON」という形式が使われています。

- JSONとは何ですか？
- なぜ広く使われていますか？

## A9.
JSONとは、データをやり取りするための書き方・形式のことです。 正式には JavaScript Object Notation の略です。
フロントエンドとバックエンドの間で、例えばカード情報を送るときに使います。

```json
{
  "title": "カード作成機能を作る",
  "description": "フォームから入力して登録できるようにする",
  "priority": "high"
}
```

このように、項目名と値の組み合わせで表すため、人間にも比較的読みやすいです。
JSONが広く使われている理由は、シンプルで軽く、JavaScriptやJavaなど多くの言語で扱いやすいからです。
今回のアプリでいうと、ReactからSpring Bootへデータを送ったり、Spring BootからReactへカード一覧を返したりするときにJSONが使われています。

## Q10. APIとは

このアプリではフロントエンドがバックエンドのAPIを呼び出してデータを取得しています。

- APIとは何ですか？
- なぜ直接データベースにアクセスしないのですか？

## A10.
APIとは、フロントエンドとバックエンドがやり取りするための「窓口」のようなものです。
今回のアプリでは、Reactが直接データベースを見るのではなく、バックエンドのAPIに対して、

カード一覧をください
新しいカードを登録してください
カードを削除してください

のように依頼します。 その依頼を受けたSpring Bootが、必要に応じてデータベースから取得したり、保存したりします。
直接データベースにアクセスしない理由は、安全性と管理のためです。 フロントエンドから直接DBに接続できてしまうと、接続情報が外に見えたり、不正にデータを操作される危険があります。
そのため、

フロントエンド → API → バックエンド → データベース

という流れにして、バックエンド側で認証、入力チェック、保存処理などを管理しています。 APIは、画面とデータベースを安全につなぐための入口だと理解しています。

## Q11. CORSとは

フロントエンド（5173）からバックエンド（8080）にアクセスする際、

「CORS」という仕組みが関係します。

CORSとは何のためにありますか？

## A11.
CORSは、別の場所からのアクセスをブラウザが安全に管理するための仕組みです。
今回のアプリでは、

フロントエンド：localhost:5173
バックエンド：localhost:8080

のように、同じ localhost でもポート番号が違うため、ブラウザ上では別の場所からのアクセスとして扱われます。
そのため、バックエンド側で

localhost:5173 からのアクセスは許可する

という設定をしておかないと、ReactからSpring BootのAPIを呼び出せません。
つまりCORSは、許可していないサイトから勝手にAPIを使われないようにするためのブラウザの安全対策だと理解しています。

## Q12. クライアントとサーバー

「クライアント」と「サーバー」とは何ですか？

このアプリの構成の中で、それぞれどれがクライアントでどれがサーバーですか？

- ブラウザ
- バックエンド
- データベース

## A12.
クライアントとは、サービスを使う側・リクエストを送る側です。 サーバーとは、リクエストを受けて処理し、結果を返す側です。
このアプリでは、基本的には以下のように考えています。

ブラウザ      = クライアント
バックエンド  = サーバー
データベース  = データを保存するサーバー

具体的には、ブラウザ上のReactが「カード一覧をください」とバックエンドにリクエストします。 バックエンドはそれを受け取り、必要に応じてデータベースから情報を取得して、ブラウザに返します。
つまり、

ブラウザ → バックエンド → データベース

という流れで動いていると理解しています。

## Q13. フロントエンドとバックエンド

「フロントエンド」と「バックエンド」の違いを説明してください。

このアプリではそれぞれ何を担当していますか？

## A13.
フロントエンドは、ユーザーが直接見る画面側のことです。 バックエンドは、画面からの依頼を受けて、データの取得や保存などを行う裏側の処理です。
このアプリでは、フロントエンドは React で作っていて、カード一覧の表示、入力フォーム、編集モーダル、ドラッグ&ドロップなどを担当しています。
バックエンドは Spring Boot で作っていて、カードの取得・作成・更新・削除などのAPI処理や、データベースへの保存を担当しています。
流れとしては、

フロントエンド → バックエンド → データベース

です。
つまり、フロントエンドは「見た目と操作」、バックエンドは「データ処理と保存の管理」を担当していると理解しています。

## Q14. 3層構造

このアプリは

- フロントエンド
- バックエンド
- データベース

の3層構造です。

なぜこのように役割を分けるのですか？

1つのプログラムですべてをやらない理由を説明してください。

## A14.
役割を分ける理由は、それぞれの担当を分かりやすくして、安全に管理しやすくするためです。
このアプリでは、

フロントエンド = 画面を表示して、ユーザーの操作を受け取る
バックエンド = 入力内容をチェックして、処理を行う
データベース = データを保存する

という役割に分けています。
もし1つのプログラムですべてをやると、画面表示、処理、データ保存が混ざってしまい、修正や確認が難しくなります。 また、フロントエンドから直接データベースを操作すると、接続情報が外に見えたり、不正にデータを操作されるリスクもあります。
そのため、

フロントエンド → バックエンド → データベース

という形にして、画面側・処理側・保存側の役割を分けています。 こうすることで、開発や修正がしやすくなり、安全性も高くなると理解しています。

## Q16. エンドポイントとURL設計

このアプリには以下のAPIがあります。

- `/api/tasks`
- `/api/tasks/{id}`

このようなURL設計の考え方を説明してください。

`{id}` の部分は何ですか？

## A16.
/api/tasks や /api/tasks/{id} は、タスクというデータに対して操作するためのAPIのURLです。
/api/tasks は、タスク全体を扱うURLです。 例えば、タスク一覧を取得したり、新しいタスクを作成したりするときに使います。

GET  /api/tasks  → タスク一覧を取得
POST /api/tasks  → 新しいタスクを作成

/api/tasks/{id} は、特定の1件のタスクを扱うURLです。 {id} はタスクを識別するための番号です。
例えば、

/api/tasks/3

なら、IDが3のタスクを指します。

GET    /api/tasks/3  → IDが3のタスクを取得
PUT    /api/tasks/3  → IDが3のタスクを更新
DELETE /api/tasks/3  → IDが3のタスクを削除

つまり、/api/tasks はタスク全体、/api/tasks/{id} は特定の1件のタスクを操作するためのURLだと理解しています。

## Q17. 同期処理と非同期処理

「同期処理」と「非同期処理」の違いを、日常生活の例えで説明してください。

なぜWeb通信では非同期処理が必要ですか？

## A17.
同期処理は、電話に近いです。 相手に電話をかけたら、相手が出て話が終わるまで、その対応に集中する必要があります。 つまり、1つの処理が終わるまで次に進めないイメージです。
非同期処理は、メールに近いです。 メールを送ったあとは、返事を待っている間に別の作業ができます。 返事が来たら、そのタイミングで内容を確認します。
Web通信では、APIからデータが返ってくるまで時間がかかることがあります。 その間に画面が止まると使いにくいため、非同期処理を使って、データ取得を待ちながらも画面操作を続けられるようにしています。

## Q18. SPA（シングルページアプリケーション）

このアプリはReactで作られたSPAです。

従来のWebサイト（ページ遷移するもの）と何が違いますか？

## A18.
SPAは、最初に1つの画面を読み込んだあと、必要な部分だけを切り替えて表示するWebアプリです。
従来のWebサイトは、ページを移動するたびにサーバーから新しいHTMLを読み込み直します。 一方、SPAではページ全体を毎回読み込み直さず、Reactが画面の一部だけを更新します。
このアプリでは、カードの作成・編集・削除などをしたときに、画面全体を再読み込みするのではなく、APIから必要なデータを取得してReact側で表示を更新しています。

## Q19. 入力チェック（バリデーション）の多重防御

入力チェックはフロントエンドとバックエンドの両方で行うべきと言われています。

なぜ片方だけでは不十分なのですか？

## A19.
入力チェックを両方で行う理由は、使いやすさと安全性の両方を守るためです。
フロントエンドの入力チェックは、ユーザーが入力した時点で「タイトルが未入力です」などをすぐ表示できるので、使いやすくなります。 ただし、フロントエンドのチェックはブラウザ側の処理なので、悪意のある操作やAPIを直接呼び出す方法で回避される可能性があります。
そのため、最終的にはバックエンド側でも必ずチェックする必要があります。 バックエンドでチェックしておけば、不正なデータがデータベースに保存されるのを防げます。
つまり、

フロントエンド = ユーザーに分かりやすく伝えるため
バックエンド = 不正なデータを保存しないため

という役割で、片方だけでは不十分だと理解しています。

## Q20. エラーハンドリング

API呼び出しが失敗した場合（ネットワーク障害やサーバーエラーなど）、

アプリはどのように対処すべきですか？

## A20.
API呼び出しが失敗した場合は、まずユーザーに失敗したことが分かるように表示する必要があります。
例えば、

データの取得に失敗しました。時間をおいて再度お試しください。

のように表示します。
何も表示されないと、ユーザーは「処理中なのか」「失敗したのか」が分かりません。 また、画面全体が止まらないようにし、可能であれば再読み込みや再実行ができるようにします。
つまり、API失敗時は、

失敗を分かりやすく伝える
画面を止めない
もう一度試せるようにする

ことが大事だと考えています。

# 第3章 データベースの基礎（8問）

## Q21. データベースとは

データベースとは何ですか？

ExcelやCSVで保存する方法と比べて何が優れていますか？

## A21.
データベースとは、アプリで使うデータを整理して保存する場所です。 このアプリでは、ユーザー情報、ボード、リスト、カードなどを保存しています。
ExcelやCSVでもデータは保存できますが、データベースの方が、アプリから安全に扱いやすいです。
例えば、データベースでは、

必要なデータだけを検索できる
複数のデータの関係を管理できる
同時に複数人が使っても壊れにくい
不正なデータが入らないように制御できる

といったメリットがあります。
ExcelやCSVは人が見るには便利ですが、Webアプリでデータを登録・更新・削除し続ける用途には、データベースの方が向いていると理解しています。

## Q22. テーブル・行・列

以下をExcelのシートに例えて説明してください。

- テーブル
- 行（レコード）
- 列（カラム）

tasksテーブルで具体例も挙げてください。

この場合、

tasksテーブル = タスクを保存するシート
1行分 = 1つのタスク情報
title列 = タスク名を入れる項目
due_date列 = 期限日を入れる項目

という理解です。
つまり、データベースはExcelのように表でデータを管理しますが、アプリから検索・登録・更新しやすいように管理されているものだと考えています。

## Q23. 主キー（PRIMARY KEY）

tasksテーブルの `id` は主キーです。

- 主キーとは何ですか？
- なぜ必要ですか？

## A23.
主キーとは、テーブルの中で1件のデータを特定するための番号です。
tasks テーブルの id が主キーの場合、同じ id は存在せず、1つのタスクを必ず区別できます。
例えば、

id: 1 → ログイン機能を作る
id: 2 → カード作成機能を作る

のように、id を見れば「どのタスクのことか」が分かります。
主キーが必要な理由は、タスクを編集・削除・取得するときに、対象を間違えないようにするためです。
例えば、

/api/tasks/1

なら、id が1のタスクを操作するという意味になります。
つまり主キーは、データを一意に識別するための目印だと理解しています。

## Q25. データの制約

tasksテーブルでは

- `NOT NULL`
- `DEFAULT 'medium'`

が設定されています。

これらは何のためにありますか？

## A25.
NOT NULL は、その項目を空のまま保存できないようにする設定です。 例えば、タスク名が空のままだと何のタスクか分からないため、title などには NOT NULL を設定します。
DEFAULT 'medium' は、値が指定されなかったときに自動で初期値を入れる設定です。 例えば、優先度を選ばずにタスクを作成した場合でも、自動で medium が入ります。
つまり、

NOT NULL = 空のデータを防ぐため
DEFAULT 'medium' = 入力がない場合の初期値を決めるため

だと理解しています。

## Q26. インデックス

インデックスとは何ですか？

本の索引に例えて説明してください。

## A26.
インデックスとは、データを早く探すための目印です。
本の索引に例えると、知りたい言葉を探すときに、最初から全部のページを読むのではなく、巻末の索引を見て「何ページに載っているか」を確認します。
データベースのインデックスも同じで、すべてのデータを1件ずつ探すのではなく、インデックスを使って目的のデータを早く見つけます。
例えば、tasks テーブルで title や due_date にインデックスを付けると、タスク名や期限日で検索するときに速くなります。
つまり、インデックスはデータ検索を効率よくするための仕組みだと理解しています。

## Q27. トランザクション

トランザクションとは何ですか？

タスクを別カラムへドラッグ移動する操作で、なぜ重要ですか？

## A27.
トランザクションとは、複数の処理をまとめて1つの処理として扱う仕組みです。 途中で失敗した場合は、最初からなかったことにできます。
例えば、タスクを別カラムへドラッグ移動するときは、

タスクの所属リストを変更する
並び順を更新する
他のタスクの順番も調整する

といった複数の更新が発生します。
もし途中でエラーが起きると、

画面では移動したように見える
でもDBでは中途半端に保存されている

という状態になる可能性があります。
そのためトランザクションを使い、すべて成功したら保存し、どこかで失敗したら元に戻すようにします。
つまり、トランザクションはデータを中途半端な状態にしないために重要だと理解しています。

## Q28. SQLとORM

- SQLとはどのようなものですか？
- ORMの役割とは何ですか？

## A28.
SQLは、データベースを操作するための言語です。 例えば、データを取得したり、登録・更新・削除するときに使います。

SELECT = データを取得する
INSERT = データを追加する
UPDATE = データを更新する
DELETE = データを削除する

ORMは、SQLを直接たくさん書かなくても、Javaのクラスやメソッドを使ってデータベースを操作できる仕組みです。
このアプリでは、JPA/HibernateがORMの役割をしています。 例えば、Card クラスを使って、cardsテーブルのデータを取得・保存できます。
つまり、

SQL = データベースに命令する言語
ORM = Javaのコードからデータベースを扱いやすくする仕組み

だと理解しています。

## Q29. バージョン管理（Git）

Gitなどのバージョン管理システムとは何ですか？

なぜ必要ですか？

使わないとどんな問題が起きますか？

## A29.
Gitなどのバージョン管理システムとは、ファイルの変更履歴を記録して管理する仕組みです。
必要な理由として、いつ・誰が・どの内容を変更したのかを確認でき、問題が起きたときに前の状態へ戻せるからです。
使わない場合、例えばコードを修正してエラーが出たときに、

どこを変更したか分からない
前の正常な状態に戻せない
複数人で作業すると上書きしてしまう

といった問題が起きます。
このプロジェクトでも、Gitを使うことで変更内容をコミットとして記録し、GitHubに保存して、PRで確認してからmainに反映できるようにしています。 つまり、Gitは安全に開発を進めるための履歴管理の仕組みだと理解しています。

## Q30. ブランチとPull Request

- ブランチとは何ですか？
- Pull Request の目的は何ですか？
- なぜ全員が main に直接変更しないのですか？

## A30.
ブランチとは、main とは別に作業するための場所です。 新しい機能追加や修正をするときに、いきなり main を変更せず、作業用ブランチで変更します。
Pull Request は、そのブランチで行った変更を main に入れてよいか確認するためのものです。 変更内容を見直したり、問題がないかチェックしてから反映できます。
全員が main に直接変更しない理由は、main は安定した状態にしておく必要があるからです。 直接変更すると、まだ動作確認していないコードや不具合が入ってしまう可能性があります。
つまり、

ブランチ = 安全に作業する場所
Pull Request = mainに入れる前の確認
main = 安定版として守る場所

だと理解しています。

## Q31. コミットメッセージの重要性

例：

`feat: ログイン機能を追加`

良いコミットメッセージを書くことがなぜ重要ですか？

## A31.
良いコミットメッセージを書くことが重要な理由は、あとから変更内容を確認しやすくするためです。
例えば、

feat: ログイン機能を追加

と書いておくと、このコミットで「ログイン機能を追加した」とすぐ分かります。
もし適当に、

修正

だけだと、あとから見たときに何を変更したのか分かりにくくなります。
良いコミットメッセージがあると、

何を変更したか分かる
なぜ変更したか追いやすい
不具合が出たときに原因を探しやすい

というメリットがあります。
つまり、コミットメッセージは未来の自分や他の人が変更履歴を理解するためのメモだと考えています。

## Q32. なぜ技術的な仕組みを理解すべきか

AI時代においてコードはAIが書けるようになりました。

それでも

- ネットワーク
- データベース
- Webアプリケーションの仕組み

を理解する必要がある理由を、あなたの考えで述べてください。

## A32.
AIがコードを書ける時代でも、仕組みを理解する必要はあると思います。
理由は、AIが出したコードが本当に正しいか、自分で判断する必要があるからです。 ネットワーク、データベース、Webアプリの流れを理解していないと、エラーが出たときに原因を切り分けられません。
例えば、画面にデータが表示されない場合でも、

フロントエンドの問題なのか
バックエンドのAPIの問題なのか
データベースに保存されていないのか
通信エラーなのか

を考える必要があります。
AIは作業を効率化してくれますが、最終的に動作確認し、問題を見つけて直すのは自分です。
そのため、AIに任せるだけではなく、基本的な仕組みを理解して、AIの回答を確認・修正できる力が必要だと考えています。
